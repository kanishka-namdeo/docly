# Local RAG Assistant Design Spec

**Date:** 2026-07-10  
**Status:** Draft  
**Author:** Design Session

## Overview

A local-first desktop RAG (Retrieval-Augmented Generation) assistant that enables users to chat with their documents using agentic retrieval patterns. The app runs locally with cloud LLM APIs for reasoning, providing document Q&A with full citation tracking and source verification.

## Core Architecture

### Desktop Application
- **Framework:** Tauri 2.x (Rust shell + web frontend)
- **Frontend:** React (chat UI, document library, settings)
- **Backend:** Python FastAPI sidecar (RAG logic, document processing)
- **IPC:** Tauri commands for frontend-backend communication

### Agentic RAG Pipeline
- **Orchestration:** LangGraph-style control loop (Python)
- **Retrieval:** LlamaIndex-compatible pipeline with Qdrant
- **Vector Database:** Qdrant embedded mode (hybrid search: BM25 + vector + RRF)
- **Embeddings:** LM Studio (local, OpenAI-compatible API)
- **LLM:** Cloud APIs (Anthropic, OpenAI, Google, or custom OpenAI-compatible)
- **Document Parsing:** Unstructured.io (advanced parsing for PDF, DOCX, Excel, Markdown, web pages)

### Production-Grade Components
- **Tracing:** Arize Phoenix (embedded mode, in-process)
- **Evaluation:** Ragas (offline quality metrics)
- **Caching:** GPTCache (semantic cache, SQLite backend)
- **Retry Logic:** tenacity (exponential backoff for API calls)

## Document Management

### Collections
- Persistent document groupings (not just folder pointers)
- Created from folders, individual files, or manual selection
- Each collection has: name, description, document count, indexing status
- Users can create/edit/delete collections in the UI

### Document Ingestion
- **Chunking Strategy:** Document-type-aware with hierarchical context
  - PDF: extract text + preserve structure (headings, pages)
  - Markdown: split on headings, preserve hierarchy
  - DOCX: extract paragraphs, respect sections
  - Excel: each sheet → table, each row → chunk with headers
  - Web pages: extract main content, split on DOM structure
- **Chunk Size:** 512-1024 tokens with 50-100 token overlap
- **Hierarchical Context:** Small chunks for retrieval, parent context (full section) for LLM

### Auto-Watch
- File system monitoring via `watchdog` library
- Debounce events (1-2s after last change before processing)
- Background worker thread for re-indexing (non-blocking)
- Document status tracking: `pending` → `indexed` or `error`
- **Behaviors:**
  - New file → parse, chunk, embed, add to Qdrant
  - File modified → remove old chunks, re-parse, re-embed, update Qdrant
  - File deleted → remove from Qdrant
  - Folder added → scan and index all files
  - Collection deleted → remove all associated documents
- **Edge Cases:**
  - Large files (>100MB) → skip or warn user
  - Unsupported file types → skip silently
  - Parse errors → mark as `error` status, show in UI
  - Temporary files (`.tmp`, `~`) → ignore

## Conversation Management

### Multiple Conversations
- Sidebar list with create/rename/delete
- Each conversation optionally targets a collection
- If no collection selected → search all documents
- If collection selected → search only that collection's documents
- Each conversation stores its own message history and retrieved context

### Context Window Management
- **Strategy:** Sliding window with summary
- Keep last 10 messages in full (recent context)
- When conversation exceeds 10 messages, summarize older messages into a single context block
- Send to LLM: `[system prompt] + [conversation summary] + [last 10 messages] + [current query + retrieved chunks]`
- Summary updates incrementally (re-summarize when new messages push old ones out)
- Summary generated by LLM (separate call, cached)
- Store summary in conversation metadata

## Agentic RAG Control Flow

### Iteration Loop
1. **Query Planning** — LLM analyzes question, decomposes into sub-queries if needed
2. **Retrieve** — Hybrid search (BM25 + vector) retrieves top-50 chunks
3. **Rerank** — Qdrant's RRF fusion ranks results
4. **Evaluate Retrieval** — LLM scores confidence (high/medium/low)
   - High confidence → proceed to generation
   - Medium confidence → refine query and re-retrieve
   - Low confidence → decompose further or escalate
5. **Generate** — LLM produces answer with footnote citations
6. **Self-Critic** — LLM validates answer against retrieved chunks
   - If unsupported → retry with different chunks
   - If supported → return answer
7. **Iteration Cap** — Max 5-6 loops to prevent infinite retries

### Retrieval Strategy
- **Hybrid Search:** BM25 (keyword) + dense vector search fused via Reciprocal Rank Fusion (RRF)
- **Top-K:** Retrieve top-50 chunks, return top-5-10 after ranking
- **Metadata Filtering:** Filter by collection_id when conversation is scoped to a collection

### Citation System
- **Style:** Footnote-style citations with clickable source cards
- Each source shown as a collapsible card with:
  - Document name
  - Page/section/chunk location
  - Relevant excerpt
  - Relevance score
- Clean answers with numbered references (like academic papers)
- Users can click to expand source details without cluttering the response

## Model Configuration

### Provider Types

#### Built-in Providers (pre-configured)
- **Anthropic:** Claude Sonnet 4.5, Claude Opus 4
- **OpenAI:** GPT-4o, GPT-4o-mini, o1, o3
- **Google:** Gemini 2.0 Flash, Gemini 2.0 Pro

#### Custom OpenAI-Compatible Providers (user-configured)
- Base URL (e.g., `http://localhost:1234/v1` for LM Studio, `https://api.groq.com/openai/v1` for Groq)
- Model name (user enters exact model ID)
- API key (optional, stored in keychain)
- Users can save multiple custom providers as presets

## API Key Storage

### Primary: System Keychain
- **Windows:** Windows Credential Manager
- **macOS:** Keychain Services (optional Touch ID integration)
- **Linux:** Secret Service (GNOME Keyring, KDE Wallet) or kernel Keyutils fallback

### Fallback: Encrypted Config File
- For headless Linux or if keychain unavailable
- AES-256 encryption with master password
- App detects keychain availability and prompts user

### Implementation
- Use `tauri-plugin-keyring-store` for frontend access
- Store API keys in frontend (React settings page)
- Retrieve in backend (Python sidecar) via Tauri command
- Zero configuration for users

## Embeddings Configuration

### LM Studio Setup (User Action Required)
- User must install LM Studio separately from https://lmstudio.ai
- User must download an embedding model (e.g., `nomic-embed-text-v1.5` or `bge-base-en-v1.5`)
- User must start LM Studio server (default: `http://localhost:1234`)
- App provides setup instructions in Settings → Embeddings

### App Behavior
- On first launch, check LM Studio connection: `GET http://localhost:1234/v1/models`
- If not connected: show setup wizard with instructions
- If connected: proceed normally

### Settings UI
```
Settings → Embeddings

LM Studio Status: [Connected ✓ / Not Connected ✗]
LM Studio URL: [http://localhost:1234] (editable)
Embedding Model: [nomic-embed-text-v1.5] (detected from LM Studio)

[Setup Instructions] button → opens modal with step-by-step guide
```

## Out-of-the-Box Functionality

### Automatic (No Setup Required)
✓ **Qdrant embedded** — starts in-process on app launch  
✓ **Arize Phoenix** — embedded tracing starts automatically  
✓ **GPTCache** — SQLite cache auto-initializes  
✓ **Unstructured.io** — document parsers work immediately  
✓ **Auto-watch** — starts when user adds first collection  
✓ **System keychain** — OS feature, no setup  
✓ **Ragas** — runs on-demand for evaluation  
✓ **tenacity** — retry logic is automatic  

### User Setup Required
- **LM Studio** — install, download embedding model, start server
- **LLM Provider** — configure API key and model selection

### First-Run Experience
1. User installs app
2. App starts, checks LM Studio connection
3. If LM Studio not running: show setup wizard
   - "Download LM Studio from https://lmstudio.ai"
   - "Download embedding model: nomic-embed-text-v1.5"
   - "Start LM Studio server (default port 1234)"
   - [Check Connection] button
4. User configures LLM provider in Settings (API key, model)
5. User adds folder to create first collection
6. App auto-indexes documents (shows progress)
7. User can start chatting

## Performance Expectations

### Latency
- **Simple queries (classic RAG path):** 1-2 seconds
- **Complex queries (agentic loop):** 4-8 seconds average
- **Multi-hop queries:** 8-15 seconds (with iteration)

### Cost
- **3-10x more tokens** than classic RAG due to agentic loops
- **Mitigation:** GPTCache semantic caching reduces repeated query costs by 30-50%
- **Iteration cap:** Max 5-6 loops prevents runaway costs

### Quality Targets (Ragas Metrics)
- Faithfulness ≥ 0.9 (no hallucinations)
- Answer relevancy ≥ 0.85 (on-topic responses)
- Context precision ≥ 0.8 (good retrieval)
- Context recall ≥ 0.85 (no missing chunks)

## Error Handling

### Document Parsing Errors
- Mark document as `error` status
- Show error icon in UI with tooltip
- User can retry indexing or remove document

### LLM API Errors
- Retry with exponential backoff (tenacity)
- Show user-friendly error message after max retries
- Log full error to Arize Phoenix for debugging

### LM Studio Connection Errors
- Show connection status in settings
- Prompt user to start LM Studio if disconnected
- Graceful degradation: app works but embeddings unavailable

### Qdrant Errors
- Embedded mode means no network errors
- Handle disk full / corruption with clear error messages
- Auto-recovery: rebuild index from documents if needed

## Observability

### Arize Phoenix Tracing
- Trace every agentic loop iteration
- Track: query planning, retrieval, evaluation, generation, self-critique
- View traces via local URL (opens in browser)
- Debug which queries fail, where retries happen
- Track token costs per query

### Ragas Evaluation
- Build 50-200 query golden set
- Run evaluation on every prompt/retrieval change
- CI gate: block deploys if metrics drop below targets
- Track drift weekly against frozen golden set

### GPTCache Metrics
- Cache hit rate
- Token savings
- Most common queries

## Security

### API Key Storage
- System keychain (OS-native encryption)
- Fallback: AES-256 encrypted config file
- Keys never logged or exposed in UI

### Document Privacy
- All processing happens locally
- Documents never leave the machine (except chunks sent to LLM API)
- User controls which documents are indexed

### Network Security
- LM Studio: localhost only (no external access)
- Cloud APIs: HTTPS only
- No telemetry or analytics (user can opt-in)

## Deployment

### Build Process
- Tauri builds native binaries for Windows, macOS, Linux
- Python sidecar bundled via PyInstaller
- Single installer per platform

### Distribution
- Windows: `.msi` or `.exe` installer
- macOS: `.dmg` installer
- Linux: `.AppImage` or `.deb`/`.rpm` packages

### Updates
- Tauri auto-update support
- User can check for updates manually or enable auto-update

### Performance
- Simple queries: < 2 seconds

## Conclusion

This design delivers a production-grade local RAG assistant with agentic retrieval capabilities. The stack balances simplicity (embedded databases, minimal external dependencies) with quality (hybrid search, agentic loops, evaluation pipeline). The out-of-the-box experience minimizes setup friction while maintaining flexibility for advanced users.

The agentic RAG pattern provides higher quality answers on complex queries at the cost of increased latency and token usage. The sliding window with summary, semantic caching, and iteration caps keep costs predictable. System keychain integration and embedded tracing provide security and observability without external infrastructure.

This architecture is ready for implementation and can be extended with future enhancements as needed.
